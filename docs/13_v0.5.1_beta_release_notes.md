# TestPlay Runner v0.5.1-beta — Release Notes

**Release date:** 2026-04-03
**Previous release:** v0.5.0-beta

---

## Summary

v0.5.1-beta is a hardening release. It introduces exit 9 to distinguish runner infrastructure failures from test results, and expands the `testplay list` scanner to detect parameterized test attributes (`[TestCase]`, `[TestCaseSource]`, `[Theory]`).

**Impact on agent-driven workflows:** agents can now distinguish "tests passed but results were lost" (exit 9) from "tests actually passed" (exit 0). The `list` scanner now reports parameterized test methods, closing the gap between what `list` detects and what `run` parses.

---

## What's New

### 1. Exit 9 — Runner system error

A new exit code distinguishes runner infrastructure failures from Unity test outcomes.

**When exit 9 is returned:**
- `Store.Save()` failure — run result JSON not persisted (agent cannot retrieve via `testplay result`)
- `SaveSummary()` failure — `summary.json` artifact not written
- `SaveManifest()` failure — `manifest.json` artifact not written

**When exit 9 is NOT returned:**
- `--compare-run` target not found — warning only, does not affect test results
- Library cache update failure — warning only, non-critical optimization

**Agent decision tree:**

| Exit | Agent action |
|------|-------------|
| 0 | Proceed — all tests passed, results saved |
| 9 | Check disk space/permissions — tests may have passed but results are lost |

The `warnings` field (string array) is included in the JSON output whenever exit 9 is returned, detailing which save operations failed.

**Scenario mode:** `exitCodeLabel(9)` returns `"runner system error"` for fast-fail error messages.

### 2. List scanner — parameterized test attributes

`testplay list` now detects three additional C# test attributes:

| Attribute | Example | Detected |
|-----------|---------|----------|
| `[Test]` | `public void BasicTest()` | ✅ (existing) |
| `[UnityTest]` | `public IEnumerator AsyncTest()` | ✅ (existing) |
| `[TestCase(...)]` | `[TestCase(1, 2, 3)]` | ✅ **new** |
| `[TestCaseSource(...)]` | `[TestCaseSource(nameof(Data))]` | ✅ **new** |
| `[Theory]` | `[Theory]` | ✅ **new** |

The scanner uses `strings.Contains` line-by-line matching (not regex), and correctly handles multi-line attribute stacking:

```csharp
[TestCase(1, 2, 3)]
[Category("Math")]       // ← skipped (additional attribute line)
public void AddTest() {} // ← detected as test method
```

### 3. Summary/manifest exit code consistency

`summary.json` and `manifest.json` artifacts are now written **after** the exit 9 override, ensuring they capture the final exit code rather than Unity's raw exit code. This means:

- If `Store.Save()` fails → `summary.json` records `exit_code: 9` (not 0)
- The agent-facing JSON (stdout) and internal artifacts are consistent

## Exit Code Table (updated)

| Code | Meaning | Agent action | Implemented |
|------|---------|-------------|-------------|
| 0 | All tests passed | Proceed | ✅ |
| 1 | Dependency error | Fix env, check `hint` | ✅ |
| 2 | Compile failure | Fix source code | ✅ |
| 3 | Test failure | Fix test logic | ✅ |
| 4 | Timeout | Check `timeout_type` | ✅ |
| 5 | Config error | Fix config file | ✅ |
| 6 | Build failure | Fix build environment | ❌ |
| 7 | Permission error | Fix path/permissions | ❌ |
| 8 | Interrupted | Retry | ✅ |
| 9 | Runner system error | Check disk/permissions | ✅ **new** |

---

## CLI Changes

No new flags or commands. Output schema gains the `warnings` field on exit 9 (additive, non-breaking).

---

## What Is NOT Supported in This Release

- **Exit 6 (build failure)** and **exit 7 (permission)** — documented but not yet returned
- **Streaming log write failure detection** — `stdout.log`/`stderr.log` write errors during Unity execution are not caught (Go `exec.Cmd` limitation); only post-execution save operations trigger exit 9
