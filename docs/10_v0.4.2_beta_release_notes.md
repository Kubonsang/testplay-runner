# TestPlay Runner v0.4.2-beta — Release Notes

**Release date:** 2026-04-02
**Previous release:** v0.4.1-beta

---

## Summary

v0.4.2-beta eliminates the Library/ cold-start bottleneck in shadow workspace mode. File copying is now parallelized (8-worker goroutine pool), and a project-local Library cache seeds subsequent shadow runs from disk instead of forcing Unity to reimport from scratch every time.

**Impact on agent-driven workflows:** the first shadow run after a cache miss still cold-starts (2-5 min), but every subsequent run seeds Library/ from cache (5-15 sec copy), saving 2-5 minutes per run. In a typical 10-iteration fix-and-test cycle, this saves 20-45 minutes of idle wait time.

---

## What's New

### 1. Parallel file copy — `copyDir` parallelized

The internal `copyDir` function — used to copy `Assets/`, `ProjectSettings/`, and Library cache into shadow workspaces — now dispatches file copies to an 8-goroutine worker pool instead of copying sequentially.

| Metric | Before (v0.4.1) | After (v0.4.2) |
|--------|------------------|-----------------|
| Copy strategy | Sequential `filepath.WalkDir` | Dirs/symlinks sequential, files parallel (8 workers) |
| File copy throughput | 1x | ~4-8x on SSD (IO-bound, not CPU-bound) |
| Context cancellation | Per-WalkDir-entry | Per-WalkDir-entry + per-worker |

Directories and symlinks are still created sequentially during the walk (filesystem ordering requirement), but regular file copies are dispatched to the worker pool. The `atomic.Value`-based first-error pattern ensures clean shutdown on failure.

### 2. Library warm cache — `.testplay/cache/Library/`

Shadow workspace runs no longer start with an empty `Library/` directory. A project-local cache stores the Library content from successful runs and seeds subsequent shadow workspaces.

**Cache lifecycle:**

```
First run (cold)          Subsequent runs (warm)
─────────────────         ──────────────────────
Library/ = empty          Library/ = copy from cache
Unity reimports (2-5 min) Unity validates (5-15 sec)
Run completes (exit 0/3)  Run completes (exit 0/3)
→ Cache Library/ to disk  → Update cache on disk
```

**Time savings per run:**

| Phase | Cold (no cache) | Warm (cached) | Saved |
|-------|----------------|---------------|-------|
| Library import | 2-5 min | 5-15 sec (copy) | ~2-5 min |
| Total `testplay run` | 3-7 min | 30-90 sec | ~2-5 min |

**Over a 10-iteration agent workflow:**

| Scenario | Cold every run | Warm cache | Saved |
|----------|---------------|------------|-------|
| 10 runs x edit_mode | 30-70 min | 5-12 min (1 cold + 9 warm) | **25-58 min** |
| 10 runs x play_mode | 40-90 min | 8-15 min (1 cold + 9 warm) | **32-75 min** |

### 3. Cache invalidation — SHA256 key

The cache key is a SHA256 hash of two files that determine Library/ content:

- `ProjectSettings/ProjectVersion.txt` — Unity editor version
- `Packages/manifest.json` — package dependencies

When either file changes (Unity version upgrade, package add/remove), the cache is automatically invalidated. The next run cold-starts and rebuilds the cache.

```
Cache key = SHA256(
    "ProjectSettings/ProjectVersion.txt" + contents +
    "Packages/manifest.json" + contents
)
```

Stored at: `.testplay/cache/cache.key`

### 4. Cache write-back — exit 0 and exit 3 only

Library cache is updated only after runs that produce a valid Library:

| Exit code | Library state | Cache action |
|-----------|--------------|--------------|
| 0 (all passed) | Valid | Update cache |
| 3 (test failure) | Valid (compile succeeded) | Update cache |
| 2 (compile error) | Incomplete | Skip |
| 4 (timeout) | Possibly corrupt | Skip |
| 8 (interrupted) | Possibly corrupt | Skip |

### 5. `--clear-cache` flag

```bash
testplay run --clear-cache
```

Removes `.testplay/cache/` before shadow workspace creation, forcing Unity to reimport from scratch. Use when:

- The cache might be corrupted after a crash
- Troubleshooting import-related test failures
- Verifying that a test passes without cached artifacts

### 6. `PrepareOptions` — extensible shadow configuration

`shadow.Prepare()` now accepts a `PrepareOptions` struct:

```go
type PrepareOptions struct {
    LibraryCacheDir string // seed Library/ from this cache directory
}
```

Zero-value `PrepareOptions{}` preserves backward-compatible behavior (empty Library/). This enables future extensions without breaking the API.

## Runtime Files (new)

| Path | Purpose |
|------|---------|
| `.testplay/cache/Library/` | Cached Library directory for shadow workspace seeding |
| `.testplay/cache/cache.key` | SHA256 hash of cache inputs; used for invalidation |

Both files are project-local under `.testplay/` (consistent with `.testplay/results/`).

---

## What Is NOT Supported in This Release

- **Cross-project cache sharing** — each project maintains its own cache
- **Incremental cache update** — Library is fully re-copied on write-back (not delta)
- **Scenario-level cache coordination** — each instance in `--scenario` mode uses the same cache independently; no per-role caching
- **Exit 6 (build failure)** and **exit 7 (permission)** — documented but not yet returned

---

## Known Limitations

| Area | Issue | Severity |
|---|---|---|
| Cache — first run still cold | First run after cache miss reimports fully; the cache only helps subsequent runs | Expected |
| Cache — full re-copy on write-back | `UpdateLibraryCache` copies the entire Library directory; no delta/incremental update | Low |
| Cache — no scenario isolation | All scenario instances share one cache; the last successful instance's Library wins | Low |
| Shadow — `Packages/` not fully isolated | `Packages/` is linked, not copied; writes propagate to source project | Low |

## Installation

```bash
# macOS/Linux
go build -o testplay ./cmd/testplay

# Windows
GOOS=windows GOARCH=amd64 go build -o testplay.exe ./cmd/testplay
```

Verify:

```bash
testplay version
# {"schema_version":"1","version":"v0.4.2-beta"}
```

---

## Quick Start

```bash
testplay check                                    # validate environment
testplay list                                     # scan for test names
testplay run                                      # single-instance run
testplay run --shadow                             # force shadow mode (warm cache)
testplay run --clear-cache --shadow               # cold start (no cache)
testplay run --scenario scenario.json             # multi-instance with ordering
testplay result                                   # review run history
```
