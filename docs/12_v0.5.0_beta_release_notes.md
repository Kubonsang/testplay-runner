# TestPlay Runner v0.5.0-beta — Release Notes

**Release date:** 2026-04-03
**Previous release:** v0.4.2-beta

---

## Summary

v0.5.0-beta strengthens the AI-facing output contract. Host crash error messages now include the dependency's exit code and failure label, parameterized NUnit test results are parsed with `parameterized_group` metadata, and an opt-in E2E test infrastructure validates the full pipeline against a real Unity installation.

**Impact on agent-driven workflows:** agents can now immediately jump to the root cause when a scenario dependency crashes (e.g., "host failed with exit 2 (compile error)") instead of misattributing the failure. Parameterized test results are individually addressable, not collapsed into opaque groups.

---

## What's New

### 1. Host exit code propagation (Scenario)

When a dependency instance exits before reaching the required `ready_phase`, the dependent instance's error message now includes the dependency's exit code and a human-readable label.

**Before (v0.4.2):**
```
dependency "host" exited before reaching phase "compiling"
```

**After (v0.5.0):**
```
dependency "host" exited with exit 2 (compile error) before reaching phase "compiling"
```

Labels for all exit codes:

| Exit | Label |
|------|-------|
| 0 | all tests passed |
| 1 | dependency error |
| 2 | compile error |
| 3 | test failure |
| 4 | timeout |
| 5 | config error |
| 6 | build failure |
| 7 | permission error |
| 8 | interrupted |
| 9 | runner system error |

### 2. Parameterized test parsing

NUnit XML `<test-suite type="ParameterizedMethod">` nodes are now recognized. Test entries inside such suites carry a `parameterized_group` field in the JSON output.

**JSON output example:**
```json
{
  "name": "AddTest(1,2,3)",
  "parameterized_group": "AddTest",
  "result": "Passed"
}
```

- `parameterized_group` is present only for parameterized tests; absent for regular tests.
- The parser treats a missing `type` attribute on `<test-suite>` as a fatal parse error — defensive against malformed XML rather than silently dropping results.

### 3. E2E test infrastructure

Opt-in integration tests that run against a real Unity installation:

```bash
UNITY_PATH=/path/to/Unity go test -tags e2e ./e2e/...
```

- Build-tagged with `//go:build e2e` — skipped by default in `go test ./...`
- Validates: NUnit XML parsing accuracy, exit code mapping, shadow workspace path remapping, Library cache seeding
- Requires a dummy Unity project in `testdata/unity-project/`

---

## CLI Changes

No new flags or commands. The `--scenario` output schema gains the `parameterized_group` field on test entries (additive, non-breaking).

---

## Output Schema Changes

| Field | Location | Change |
|-------|----------|--------|
| `parameterized_group` | `tests[].parameterized_group` | New (string, optional) — present only for parameterized tests |
| `orchestrator_errors` | Scenario output | Error messages now include exit code and label |

---

## What Is NOT Supported in This Release

- **Exit 6 (build failure)** and **exit 7 (permission)** — documented but not yet returned
- **Exit 9 (runner system error)** — added in v0.5.1
- **`list` scanner parameterized test detection** — added in v0.5.1
