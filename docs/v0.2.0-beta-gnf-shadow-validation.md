# FastPlay v0.2.0-beta Validation Report — GNF_ Shadow PlayMode Smoke

> Status: DONE

This document records a real-project validation run for `fastplay v0.2.0-beta`
against the `GNF_` Unity project with the Unity Editor already open.
The goal was to verify that the new Shadow Workspace path is usable by an AI
agent on a non-trivial project, not only on the bundled smoke fixture.

## Validation Summary

**Result:** PASS

The following flow completed successfully on a real Unity project:

1. Build `fastplay` from current `main`
2. Validate environment with `fastplay check`
3. Discover the target test with `fastplay list`
4. Run a PlayMode movement smoke test while the Unity Editor is open
5. Confirm successful JSON output, status snapshot, artifacts, and event log
6. Re-run the same test to confirm that the persisted Shadow `Library/` cache
   materially reduces execution time

This validation demonstrates that `v0.2.0-beta` is usable for:

- a real Unity project
- PlayMode execution
- editor-open automatic Shadow backend selection
- run artifact generation
- status/event reporting
- practical iterative re-run workflow for an AI agent

## Scope Of This Validation

This report validates a **single, concrete scenario**:

- Project: `GNF_`
- Platform: `play_mode`
- Test target: character movement smoke test
- Editor state: **open** (`Temp/UnityLockfile` present)
- Backend selection: **automatic Shadow Workspace**

This report does **not** claim that every v0.2 feature or every Unity project
configuration has been validated. See [Not Validated In This Report](#not-validated-in-this-report).

## What Was Tested

### Target Test

File:

- `GNF_/Assets/Tests/PlayMode/CodexMovementSmokeTest.cs`

Target test case:

- `CodexMovementSmokeTest.TestPlayer_MovesRight_InPlayMode`

### Test Behavior

This test creates a capsule GameObject in PlayMode, attaches a simple movement
component, advances for 20 frames, and asserts that the object moved to the
right on the X axis.

In other words, this is a minimal but real PlayMode runtime smoke test for:

- PlayMode execution itself
- scene object creation
- `MonoBehaviour.Update()`
- frame progression over multiple ticks
- runtime movement assertion

## Evidence That Shadow Backend Was Actually Used

This was not a Direct Batch run disguised as success. The following evidence
shows that the Shadow backend was exercised:

1. `Temp/UnityLockfile` existed before execution
2. `.fastplay-shadow/` was created under the project root
3. `.gitignore` was auto-patched with:

   ```text
   .fastplay-shadow/
   ```

4. The PlayMode `results.xml` for the second run references a test assembly
   under the Shadow `Library/` path:

   ```xml
   fullname="/Users/gubonsang/Desktop/Dev/GNF_/.fastplay-shadow/Library/ScriptAssemblies/GNF.Tests.PlayMode.dll"
   ```

Taken together, these confirm that the Editor-open automatic fallback path was
actually used.

## What This Validation Proves

This validation provides concrete evidence that `fastplay v0.2.0-beta` can:

- run against a real Unity project, not only a bundled fixture
- execute a targeted PlayMode test successfully
- auto-switch to Shadow Workspace when the Editor is open
- persist and reuse a Shadow `Library/` cache across runs
- return stable JSON output for an AI agent
- write status and event artifacts during execution
- preserve a practical iterative workflow, with the second run dramatically
  faster than the first

## Conclusion

`fastplay v0.2.0-beta` successfully completed an **editor-open, Shadow-backed,
real-project PlayMode smoke validation** on `GNF_`.

The first run paid the expected initial import cost, and the second run
demonstrated the practical payoff of the persisted Shadow `Library/` cache.
For the tested scenario, the tool behaved as an AI agent-friendly runner:

- discoverable
- executable
- inspectable
- reproducible

For `v0.2.0-beta`, this is strong evidence that the Shadow Workspace feature is
ready to be presented as a completed validation milestone, provided its known
limitations remain clearly documented.
