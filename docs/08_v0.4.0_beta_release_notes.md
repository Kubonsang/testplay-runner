# TestPlay Runner v0.4.0-beta — Release Notes

**Release date:** 2026-04-01
**Previous release:** v0.3.0-beta

---

## Summary

v0.4.0-beta delivers the Orchestrator: Host/Client startup ordering via Go channels, per-instance status polling, and structured orchestration error reporting. Dependent instances wait for their dependency to reach a configurable ready phase before starting.

Additionally, v0.4.1-beta (this release) fixes two critical orchestration safety issues: circular dependency detection and fast-fail on dependency crash.

---

## What's New

### 1. Per-instance status polling (P1 resolved)

Each instance in `--scenario` mode now writes its own status file:

```
testplay-status-host.json
testplay-status-client.json
```

Agents can poll per-role status files independently during a scenario run. Same schema as single-mode `testplay-status.json`.

### 2. Host ready gating — `depends_on` / `ready_phase` / `ready_timeout_ms`

Scenario JSON now supports startup ordering:

```json
{
  "schema_version": "1",
  "instances": [
    { "role": "host",   "config": "host/testplay.json" },
    { "role": "client", "config": "client/testplay.json",
      "depends_on": "host",
      "ready_phase": "compiling",
      "ready_timeout_ms": 10000 }
  ]
}
```

| Field | Default | Description |
|---|---|---|
| `depends_on` | _(none)_ | Role name this instance waits for |
| `ready_phase` | `"compiling"` | Phase the dependency must reach before this instance starts |
| `ready_timeout_ms` | `30000` | Maximum wait time in milliseconds |

**How it works internally:**
- Go channels (`readyCh` / `doneCh`) for synchronization — no disk polling
- `ReadyNotifier` wraps `StatusWriter`; closes `readyCh` when target phase is written
- `RunScenario` uses `select` with `<-readyCh`, `<-doneCh`, `<-time.After()`, `<-ctx.Done()`

### 3. `orchestrator_errors` in scenario JSON output

When a dependency wait fails, the scenario output includes a structured error array:

```json
{
  "schema_version": "1",
  "exit_code": 4,
  "orchestrator_errors": [
    "instance \"client\" timed out waiting for \"host\" to reach phase \"compiling\" (10000ms)"
  ],
  "instances": [ ... ]
}
```

The field is absent when no orchestration errors occurred.

### 4. Scenario validation hardening

`Load()` now validates:

| Check | Error |
|---|---|
| Duplicate role names | `role "host" is not unique` |
| Unknown `depends_on` reference | `depends_on "foo" references unknown role` |
| Self-dependency | `depends_on "host" cannot depend on itself` |
| Circular dependency (A->B->A) | `circular dependency detected involving role "host"` |

### 5. Fast-fail on dependency crash

If a dependency instance exits before reaching its `ready_phase` (e.g., compile error), dependent instances are unblocked immediately instead of waiting the full `ready_timeout_ms`. The orchestrator reports:

```
instance "client": dependency "host" exited before reaching phase "compiling"
```

## Scenario JSON Schema (v0.4)

```json
{
  "schema_version": "1",
  "instances": [
    {
      "role": "host",
      "config": "host/testplay.json"
    },
    {
      "role": "client",
      "config": "client/testplay.json",
      "depends_on": "host",
      "ready_phase": "compiling",
      "ready_timeout_ms": 10000
    }
  ]
}
```

All three ordering fields (`depends_on`, `ready_phase`, `ready_timeout_ms`) are optional. Instances without `depends_on` start immediately.

---

## What Is NOT Supported in This Release

- **DAG with multiple dependencies per instance** — `depends_on` is a single string, not an array. Each instance can depend on at most one other instance.
- **NGO/Mirror network harness** — no built-in network test framework integration
- **Exit code 6 (build failure)** and **exit code 7 (permission error)** — documented but not yet returned

---

## Known Limitations

| Area | Issue | Severity |
|---|---|---|
| Scenario — Library cold start | `Library/` starts empty each run per instance; expect clean-build-equivalent startup time | Low |
| Scenario — per-instance status only | No scenario-level aggregate status file; agents must poll per-role files | Low |
| Scenario — host crash timeout | Dependent instances fast-fail immediately on host crash, but the error message does not include the host's exit code or failure details | Low |
| Shadow — `Packages/` not fully isolated | `Packages/` is linked, not copied; writes propagate to source project | Low |

## Installation

```bash
# macOS/Linux
go build -o testplay ./cmd/testplay

# Windows
GOOS=windows GOARCH=amd64 go build -o testplay.exe ./cmd/testplay
```

Verify:

```bash
testplay version
# {"schema_version":"1","version":"v0.4.0-beta"}
```

## Quick Start

```bash
testplay check                                    # validate environment
testplay list                                     # scan for test names
testplay run                                      # single-instance run
testplay run --scenario scenario.json             # multi-instance with ordering
testplay result                                   # review run history
```
