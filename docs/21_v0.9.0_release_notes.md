# TestPlay Runner v0.9.0 ‚Äî Release Notes

**Release date:** 2026-04-24
**Previous release:** v0.8.0

---

## Summary

v0.9.0 ‚Äî "Network Primitives" ‚Äî adds the framework-agnostic plumbing
that lets multiple Unity instances in a `--scenario` run exchange
arbitrary messages and lets agents reason about cross-instance failures
without scraping logs.

The motivating gap: v0.6's scenario mode only supported a unidirectional
"host ready" signal via Go channels. There was no way for the host to
emit anything beyond "I reached phase X", no way for the client to send
anything back, and no way for an agent inspecting a failed scenario to
know what the failed instance last received from its peer.

v0.9 closes that gap with a single, simple primitive: a per-scenario
NDJSON file every instance can append to and poll. Everything else ‚Äî
broadcast, peer-to-peer, structured payloads, last-message correlation,
forensic timeline integration ‚Äî is built on top of that one file.

**Impact on agent-driven workflows:** agents can now drive multiplayer
test scenarios end-to-end. When a scenario fails, the output JSON tells
them exactly which messages each instance saw and which it sent, so
"why did the client time out?" becomes "the client's last received
from the host was seq=3 kind=server_started ‚Äî host never sent
player_joined" rather than a guessing game.

No backward-incompatible changes ‚Äî all new fields are additive, and IPC
capture is opt-in via the simple presence of the env var (instances
that ignore `TESTPLAY_IPC_BUS` continue to work exactly as before).

## What's New

### 1. Scenario-scoped IPC bus (`internal/ipc/`)

Every `--scenario` run now creates an empty NDJSON file at:

```
<project>/.testplay/ipc/<scenario_run_id>/bus.ndjson
```

The absolute path is injected into every instance's process environment
as `TESTPLAY_IPC_BUS`. User test code in any language reads the env var
and either appends NDJSON lines (to send) or polls the file (to receive).

**Message schema:**

```json
{"seq": 1, "ts": "2026-04-24T13:00:05Z", "from": "host", "to": "*", "kind": "ready", "payload": {"port": 7777}}
```

- `from` ‚Äî sender role
- `to` ‚Äî recipient role, or `"*"` for broadcast
- `kind` ‚Äî application-defined event name
- `payload` ‚Äî optional, arbitrary JSON
- `seq` ‚Äî sender-local monotonic counter; `(from, seq)` is the unique key

Atomic-append discipline mirrors the existing `events.ndjson` pattern:
`O_APPEND` + per-writer mutex on the Go side, OS-level atomic concat
within `PIPE_BUF` (~4096 bytes) for cross-process writes.

**Go-side helpers** (`internal/ipc`):
- `BusWriter` ‚Äî test utility for fixtures; production users append from C# directly
- `PollingReader` ‚Äî tails the bus file at a configurable interval, classifies events as `send` (own role originated) or `recv` (addressed to us or broadcast), tolerates missing file and malformed lines
- `Accumulator` + `RunReaderInto` ‚Äî thread-safe collection helper used by the scenario runner

### 2. Cross-instance failure correlation (`internal/scenario/runner.go`)

Each instance gets its own polling reader for the entire scenario
lifetime ‚Äî started before `depends_on` waits begin, so messages
arriving during dep-wait are not missed.

When the orchestrator detects that a dependency exited before
signaling its ready phase, the resulting `orchestrator_errors` entry is
enriched with the last message the waiting instance saw from that
dependency:

```
instance "client": dependency "host" exited with exit 2 (compile error) before reaching phase "compiling". "client" last received from "host": seq=1 kind="boot"
```

If the waiting instance never received anything from the dependency,
the suffix is omitted ‚Äî so "no correlation suffix" is itself a signal.

### 3. Scenario output JSON (`cmd/testplay/run.go`)

Top-level: a new `scenario_run_id` field, separate from the
per-instance `run_id`s that runsvc still emits. Used to correlate
`<project>/.testplay/ipc/<scenario_run_id>/` with the rest of a run.

Per instance, two new fields when IPC was active:

```json
{
  "role": "client",
  "ipc_messages": [
    {"seq": 1, "ts": "...", "from": "host", "to": "*", "kind": "ready"},
    {"seq": 2, "ts": "...", "from": "client", "to": "host", "kind": "joined"}
  ],
  "ipc_summary": {
    "sent_count": 1,
    "received_count": 1,
    "last_sent":     {"seq": 2, "to": "host", "kind": "joined"},
    "last_received": {"seq": 1, "from": "host", "kind": "ready"}
  }
}
```

`ipc_messages` is the full ordered list of everything this instance
sent or received (filtered from the shared bus by role). `ipc_summary`
gives agents quick triage without parsing the array.

### 4. `events.ndjson` IPC integration (`internal/status/events.go`)

After a scenario completes, captured IPC events are appended to each
instance's per-run `events.ndjson` so a single role timeline contains
both Unity phase transitions and IPC traffic:

```jsonl
{"event":"run_started","run_id":"...","timestamp":"2026-04-24T10:00:00Z"}
{"event":"ipc_recv","timestamp":"2026-04-24T10:00:05Z","ipc_seq":1,"ipc_kind":"ready","ipc_peer":"host"}
{"event":"ipc_send","timestamp":"2026-04-24T10:00:07Z","ipc_seq":1,"ipc_kind":"joined","ipc_peer":"host"}
{"event":"run_finished","timestamp":"2026-04-24T10:00:08Z","exit_code":0}
```

`status.Event` gains optional `IpcSeq` / `IpcKind` / `IpcPeer` fields
(`omitempty` ‚Äî invisible on non-IPC entries). `EventLog.Append` was
also updated to preserve a caller-supplied `Timestamp` instead of
overwriting with `time.Now()`, so IPC events keep their original
send/recv times rather than the moment of disk write.

### 5. Retention + .gitignore (`cmd/testplay/run.go`)

The IPC bus tree shares the run-id format used by `.testplay/runs/`,
so `artifacts.Store.Prune` works directly on `.testplay/ipc/`. Each
scenario completion runs the same `retention.max_runs` policy on both
trees in a single pass ‚Äî no new pruning code, no risk of policy drift.

`.testplay/ipc/` is added to the project `.gitignore` on first scenario
run, reusing `shadow.EnsureIgnored` for the same non-fatal append
discipline as the shadow workspace excluder.

### 6. Shared run-ID generator (`internal/runid/`)

Refactor: `generateRunID` moved from `internal/runsvc/runid.go` to
`internal/runid.Generate`. The `internal/runid` package already owned
the format pattern and `IsValid` validator; co-locating generation
eliminates the runsvc-internal helper and lets scenario code generate
`scenarioRunID` without depending on runsvc.

## Schema Changes

| File / Field | Type | Notes |
|---|---|---|
| Scenario JSON top-level: `scenario_run_id` | string | Always present; new in v0.9 |
| `instances[].ipc_messages` | array | Present only when IPC was active (i.e. user code wrote at least one message visible to this instance) |
| `instances[].ipc_summary` | object | Present alongside `ipc_messages`; counts + last_sent + last_received |
| `orchestrator_errors[]` (entries) | string | When a dep crash is detected, may now include `"X" last received from "Y": seq=N kind=K` suffix |
| `events.ndjson` (entries) | object | New events `ipc_send` / `ipc_recv` with optional `ipc_seq` / `ipc_kind` / `ipc_peer` |
| `<project>/.testplay/ipc/<scenario_run_id>/bus.ndjson` | file | New file; auto-created at scenario start |
| Env var `TESTPLAY_IPC_BUS` | path | Injected into every instance unless user already set it |

All additions are additive; agents that ignore the new fields continue
to work unchanged. Scenarios whose user code never touches the bus get
empty `ipc_messages` and zero counts in `ipc_summary` (or those fields
are simply absent).

## Out of Scope (planned for v1.0 or later)

- **Framework-specific helpers** (NGO `NetworkManager` bootstrap, `[NetworkTest]` attribute, `await Network.WaitForClientConnected()`) ‚Äî v1.0
- **Real-time push** (SSE / websockets) ‚Äî permanent non-goal; identity lock
- **Bidirectional RPC** ‚Äî single-direction only; users compose two messages for request/response
- **Single-mode IPC** ‚Äî `testplay run` without `--scenario` does not inject `TESTPLAY_IPC_BUS`
- **Other network frameworks** (Mirror, Photon, FishNet) ‚Äî v1.x, gated on user demand
- **Editor attach mode** ‚Äî see [docs/20_v0.9_measurement_note.md](20_v0.9_measurement_note.md); 34% ratio at maintainer's profile keeps this off the roadmap

## Upgrade Notes

No migration steps required for existing users.

To take advantage of v0.9:

- **Add `--scenario`-mode tests** that read `TESTPLAY_IPC_BUS` and append messages. Even appending a single `kind: "started"` line gives you the failure correlation benefit.
- **Branch on `scenario_run_id`** when correlating output JSON with `.testplay/ipc/<id>/bus.ndjson` for forensics.
- **Read `instances[].ipc_summary` first**, fall back to `ipc_messages` for detail. Summary fields are designed for fast triage.
- **Watch `orchestrator_errors`** for the new last-received correlation suffix when diagnosing why a dependent fast-failed.
