# v0.9 사전 측정 — shadow warm-cache 임계값 가이드

**상태:** 측정 미실시. 본인 프로젝트(예: `~/Desktop/Dev/map`)에서 1회만 측정하면 됨.

**목적:** "에디터 attach 모드를 v1.x 로드맵에 올릴 것인가"를 데이터로 결정하기 위한 게이트.

---

## 측정 절차

`testplay run --shadow` 실행 시간을 다음 조건으로 측정한다 (모두 EditMode 기준).

### 단계 1 — Cold shadow

```bash
# 캐시 비우고 1회 측정
testplay run --shadow --clear-cache
# 출력의 manifest.json에서 started_at, finished_at 차이 확인
```

또는 `time` 사용:

```bash
time testplay run --shadow --clear-cache > /dev/null
```

기록할 값: **cold_seconds** (초 단위)

### 단계 2 — Warm shadow (5회 평균)

```bash
for i in 1 2 3 4 5; do
  time testplay run --shadow > /dev/null
done
```

5회 실행 시간을 평균. 첫 1회는 이미 단계 1에서 cold로 측정했으므로 별도로 5회 추가 실행.

기록할 값: **warm_avg_seconds**

### 단계 3 — 모델 한 턴 시간 측정

평소 작업 흐름의 "한 턴(에이전트 응답 한 사이클)" 평균 시간 측정. Claude Code 사용 중이라면 메시지 간 타임스탬프로 추정 가능. 외부 측정이 어려우면 다음 기본값 사용:

| 모델 | 기본 추정 (한 턴) |
|---|---|
| Opus 4.6 (thinking) | 60–120s |
| Sonnet 4.6 | 20–40s |
| Haiku 4.5 | 5–10s |

기록할 값: **turn_seconds**

---

## 결정 기준

비율 = `warm_avg_seconds / turn_seconds`

| 비율 | 의미 | 액션 |
|---|---|---|
| < 20% | shadow는 노이즈 안 | attach 모드 v1.x 로드맵 **불필요**. v0.9/v1.0 NGO에 집중. |
| 20–50% | 보임 | 정체성 락은 유지. 단, v1.x 로드맵 후보로 메모. shadow 자체 최적화(parallel copy 강화, ProjectSettings hardlink) 먼저 시도. |
| > 50% | 병목 | v1.x에서 **attach 마일스톤 신규 검토**. 단, 제안 단계에서 "에디터 상태 오염"·"Unity 버전별 TestRunnerApi 분기"·"두 백엔드 유지보수 비용" 세 가지 비용을 명시. |

**임계값을 보기 전에 attach 모드를 로드맵에 올리지 않는다 — CLAUDE.md의 Identity anchor 원칙.**

## 참고

- v0.4.2의 Library 워밍 캐시가 이미 cold-start를 크게 줄였음. cold가 30초+ 라면 v0.4.2 회귀 가능성 의심.
- `--scenario` 모드는 cache write-back을 건너뛰므로(현재 의도된 동작) 측정 시 single-mode (`testplay run`) 쓸 것.
- 측정 결과가 어느 쪽이든, 결정의 근거를 이 문서에 남겨야 미래의 본인 또는 다른 메인테이너가 같은 논의를 반복하지 않음.
