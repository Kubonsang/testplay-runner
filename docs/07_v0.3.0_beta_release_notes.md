# TestPlay Runner v0.3.0-beta — Release Notes

**Release date:** 2026-04-01
**Previous release:** v0.2.0-beta

---

## Summary

v0.3.0-beta delivers the Multi-Instance Core: the ability to run multiple Unity processes simultaneously from a single command, with results aggregated into a single JSON output. It also resolves four P1 infrastructure blockers that were required before multi-instance execution could be safe.

Additionally, the CLI binary and all user-facing identifiers have been renamed from `fastplay` → `testplay` to match the repository name (`testplay-runner`).

---

## Breaking Changes

### CLI renamed: `fastplay` → `testplay`

All user-facing identifiers have changed:

| Before | After |
|---|---|
| Binary: `fastplay` | Binary: `testplay` |
| Config: `fastplay.json` | Config: `testplay.json` |
| Status file: `fastplay-status.json` | Status file: `testplay-status.json` |
| Artifact dir: `.fastplay/` | Artifact dir: `.testplay/` |
| Shadow dir: `.fastplay-shadow-<id>/` | Shadow dir: `.testplay-shadow-<id>/` |

**Migration:** rename your `fastplay.json` to `testplay.json`. Existing result files in `.fastplay/results/` are no longer visible to `testplay result` — copy them to `.testplay/results/` if needed.

---

## What's New

### 1. `testplay run --scenario <file>` — Multi-Instance Concurrent Execution

Run multiple Unity instances simultaneously from a single scenario JSON file:

```json
{
  "schema_version": "1",
  "instances": [
    { "role": "host",   "config": "host/testplay.json" },
    { "role": "client", "config": "client/testplay.json" }
  ]
}
```

```bash
testplay run --scenario scenario.json
```

All instances run concurrently. Output is a single aggregated JSON:

```json
{
  "schema_version": "1",
  "exit_code": 3,
  "instances": [
    { "role": "host",   "run_id": "...", "exit_code": 0, "passed": 12, ... },
    { "role": "client", "run_id": "...", "exit_code": 3, "failed": 1, "tests": [...], ... }
  ]
}
```

`exit_code` at the top level is the maximum exit code across all instances.

### 2. Unique runID (P1)

runID format changed from `YYYYMMDD-HHMMSS` (1-second granularity, collision-prone) to `YYYYMMDD-HHMMSS-xxxxxxxx` (8-char crypto-random hex suffix). Concurrent runs no longer overwrite each other's result files.

Old format result files remain readable — `testplay result` accepts both formats.

### 3. `--config` flag (P1)

```bash
testplay --config /path/to/testplay.json run
```

Config path is no longer hardcoded to `testplay.json` in cwd. Enables multi-instance orchestration where each instance uses a different config.

### 4. Per-run shadow isolation (P1)

Shadow workspace is now scoped to the run: `.testplay-shadow-<runID>/` instead of a shared `.testplay-shadow/`. Parallel `testplay run` invocations against the same project are now safe. The directory is deleted automatically after each run.

### 5. Exit 8 for signal interruption (P1)

SIGINT/SIGTERM now returns exit 8 (`interrupted`) instead of exit 4 (`timeout`). Agents can now distinguish between timeout and user interruption.

## What Is NOT Supported in This Release

- **Host/Client startup ordering** — all instances start simultaneously; there is no mechanism to wait for Host to be ready before starting Client
- **Per-instance status polling** — `testplay-status.json` is not written in `--scenario` mode; agents cannot observe progress during a scenario run
- **NGO/Mirror network harness** — no built-in network test framework integration
- **Exit code 6 (build failure)** and **exit code 7 (permission error)** — documented but not yet returned

---

## Known Limitations

| Area | Issue | Severity |
|---|---|---|
| Scenario — cold start per instance | `Library/` starts empty each run; each instance pays full reimport cost. Expect clean-build-equivalent startup time per instance in `--scenario` mode. | Low |
| Scenario — no status polling | `testplay-status.json` is not written in `--scenario` mode. No progress visibility during multi-instance runs. | Medium |
| Shadow — `Packages/` not fully isolated | `Packages/` is linked, not copied. Unity writes to `Packages/` propagate to the source project. | Low |
| Shadow — stale lockfile | A stale `Temp/UnityLockfile` after an unclean Unity exit triggers unnecessary shadow overhead. | Low |

## Installation

```bash
# macOS/Linux
go build -o testplay ./cmd/testplay

# Windows
GOOS=windows GOARCH=amd64 go build -o testplay.exe ./cmd/testplay
```

Verify:

```bash
testplay version
# {"schema_version":"1","version":"v0.3.0-beta"}
```

---

## Quick Start

```bash
testplay check                          # validate environment
testplay list                           # scan for test names
testplay run                            # single-instance run (unchanged)
testplay run --scenario scenario.json   # multi-instance run
testplay result                         # review run history
```
